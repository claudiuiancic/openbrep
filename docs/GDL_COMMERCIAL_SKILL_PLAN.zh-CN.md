# OpenBrep Pro 层 GDL 商业化 Skill 开发计划

日期：2026-05-06

## 背景

本轮 Free 层 GDL 知识库校准已经完成，核心目标是让 OpenBrep 在公开基础知识、常用 GDL 命令、HSF 参数结构、2D/3D 表达和典型物件 archetype 上不再依赖未经校验的 LLM 推断。

下一步 Pro 层不应只是增加更多通用对象。通用床、桌、椅、基础柜体、洁具和装饰模型已经能从 Archicad 自带库、BIMcomponents、BIMobject 或第三方图库获得。OpenBrep Pro 的价值应来自“专业物件 Skill + 企业/厂家知识包 + 工程交付链路”。

## 外部生态信号

- Graphisoft GDL Center 将 GDL 的典型用途定义为定制请求、办公室标准物件库、厂家产品集合，以及增强 Archicad 自带对象。
- Graphisoft Library Part Maker 证明官方也在推动“不直接写 GDL 但生成 GDL-based Library Part”的方向，并重点支持 Generic Object、Door、Window、Skylight、Lamp、MEP Library Parts。
- ArchiGDL 等服务商面向厂家、建筑师、工程师提供 Archicad GDL 对象开发，强调厂家产品数字化、企业工作流、质量控制、清单和维护。
- Central Innovation / Ci Tools 的商业资料显示门窗、清单、厂家尺寸约束、多面板组合、合规表达、窗帘/框架/计算对象等都是真实付费场景。
- Graphisoft 社区反馈显示，Archicad 用户痛点不是“完全没有对象”，而是缺少本地化、厂家化、可维护、可清单、可企业标准化的原生 GDL 对象。

## Pro 层定位

Free 层解决：

- 可靠的 GDL 语法和命令知识。
- HSF-native 项目读写。
- 常规对象生成、预览、编译和基础修复。
- 基础 archetype：书架、桌子、普通柜体、普通门窗视觉对象等。

Pro 层解决：

- 垂直类别的专业 Skill。
- 企业标准、厂家规格、地区规范、清单字段和图纸表达。
- 可维护的参数体系、2D/3D 双表达、BOM/清单、合规校核。
- 从用户目标自动推导 GDL 物件的几何形态、参数表、默认尺寸、图纸表达和工程交付约束。

## 高价值 Pro 类别优先级

| 优先级 | 类别 | 商业价值 | OpenBrep Pro 能力 |
|---|---|---|---|
| P1 | 门窗系统 / 幕墙开口系统 | 参数复杂，清单强依赖，厂家规格多，合规要求多 | 开启方式、分格、五金、玻璃、门窗编号、2D 符号、清单字段、厂家尺寸约束 |
| P1 | 企业制图标准物件 | 每个事务所标准不同，复用频率高，能直接节省制图时间 | 图签、索引、详图编号、标注、标签、图例、房间/门窗/材料表达标准 |
| P1 | 厂家产品 GDL 化 | 厂家有进入设计师 BIM 工作流的商业动机 | 型号参数化、规格约束、材质、BOM、维护升级包、官网分发包 |
| P1 | 清单 / 计算 / 校核对象 | 直接减少统计错漏，接近工程交付价值 | 面积、数量、门窗表、材料表、构件统计、Zone 关系、自动文本 |
| P2 | 橱柜 / 定制家具 / 木作 | 高定制、高复用、图纸和报价关联强 | 柜体、门板、五金、踢脚、台面、开孔、报价/清单字段 |
| P2 | 详图节点 / 构造做法库 | 企业沉淀价值高，不是通用图库可替代 | 参数化节点、2D+3D 联动、材料层、文字说明、局部详图标准 |
| P2 | MEP / 洁具 / 灯具 / 设备 | 厂家库多，但 Archicad 原生可编辑和清单表达不足 | 设备参数、接口点位、检修空间、型号清单、2D 表达 |
| P2 | 结构/围护/龙骨/框架 | 施工图和统计价值高 | 墙体龙骨、幕墙副框、木结构/轻钢框架、构件间距、数量统计 |
| P3 | 景观 / 场地 / 城市家具 | 通用模型多，但规范化表达不足 | 停车位、树池、栏杆、铺装、坡道、无障碍构件 |
| P3 | 迁移 / 版本维护 Skill | 老 GDL 对象维护痛点明显 | 参数迁移、GUID/版本策略、旧库升级、新旧对象兼容检查 |

## Skill 包标准结构

每个 Pro Skill 至少包含：

- `archetype.md`：对象类别、适用范围、边界条件、默认生成策略。
- `param_schema.md`：参数命名、单位、默认值、可选项、约束关系。
- `gdl_patterns.md`：3D 建模、2D 符号、参数脚本、UI 脚本、清单字段的推荐写法。
- `drafting_rules.md`：平面、立面、剖面、标注、线型、填充、比例/MVO 表达规则。
- `business_rules.md`：厂家型号、企业标准、地区规范、合规/尺寸/材料限制。
- `compile_checks.md`：生成后的静态检查、编译检查、预览检查和失败修复策略。
- `examples/`：可编译 HSF 样例、用户需求样例、变体样例。
- `anti_patterns.md`：禁止或高风险写法，例如简单家具误用 `SWEEP/REVOLVE/GROUP`，普通视觉门窗伪装成 host-aware 门窗。

## 建议实施路线

### 第一阶段：Pro Skill 基础设施

目标：让 OpenBrep 能区分 Free 基础知识和 Pro 垂直 Skill，并在生成前自动选择相关 Skill。

- 建立 `.openbrep/knowledge/` 与 Pro 导入知识的统一索引模型。
- 为 Pro Skill 增加 `category`、`license_tier`、`applicable_object_types`、`required_checks` 元数据。
- 在知识检索阶段加入“对象类别识别 -> Skill 召回 -> 生成上下文编排”。
- 保持 Free 层可用，不让 Pro 知识污染基础生成。

### 第二阶段：Pro Door/Window System Skill

目标：选择最强商业信号类别做第一套 Pro Skill。

- 建立门窗对象分类：单扇、双扇、推拉、折叠、上悬、百叶、多面板组合、门窗套、侧窗、亮窗。
- 建立参数体系：宽高、框厚、扇数、开启方式、玻璃、五金、编号、厂家型号、清单字段。
- 明确普通视觉对象和真正 Door/Window subtype 的边界。
- 输出至少 3 个可编译 HSF 样例。

### 第三阶段：Pro Enterprise Documentation Skill

目标：把“企业制图标准”变成高复用知识包。

- 图签、详图索引、门窗标签、材料标签、房间/Zone 文本、图例对象。
- 支持事务所默认参数、命名规范、图层/线型/填充约定。
- 强化 `REQUEST`、全局变量、标签 owner 关系、Zone 关系相关知识。

### 第四阶段：Pro Cabinet / Millwork Skill

目标：验证 OpenBrep 对定制类物件的复杂参数化能力。

- 柜体模块：地柜、吊柜、高柜、开放格、抽屉、门板、层板、踢脚、台面。
- 业务规则：板厚、封边、五金、开孔、模块宽度、组合限制。
- 输出 2D 平面表达、3D 模型、材料/数量字段。

### 第五阶段：Manufacturer Product Skill

目标：服务厂家和产品目录 GDL 化。

- 从产品规格表抽取参数、型号、尺寸、材质、颜色、性能字段。
- 生成可分发 HSF/GSM 项目和维护清单。
- 建立版本升级和旧型号迁移策略。

## 风险

- Pro Skill 容易变成“资料堆积”，必须以可编译样例和生成效果验证。
- 厂家资料可能有版权和授权限制，不能直接内置未经许可的商业规格。
- 真正 Door/Window/MEP subtype 牵涉 Archicad host 行为，不能用普通几何对象冒充。
- 企业标准高度差异化，必须支持项目级/客户级知识隔离。
- 过早做太多类别会分散质量，应优先做门窗和企业制图两个高价值闭环。

## 下一步建议

1. 先实现 Pro Skill 元数据和召回结构。
2. 选择 `Pro Door/Window System Skill` 作为第一套样板。
3. 每个 Skill 必须包含至少一个可编译 HSF 项目和对应失败案例。
4. 用 OpenBrep 的错题本机制反哺 Pro Skill 的 `anti_patterns.md`。
5. 等第一套 Pro Skill 能稳定生成可编译对象后，再扩展橱柜、厂家产品和清单计算类。
